THE SCENARIO

두 가지 결, 하나의 비서

아침에는 에이전트가 먼저 보고하고(Push), 그 이후엔 CEO가 부르면 종합해서 답합니다(Pull). 같은 Teams 대화방 안에서 자연스럽게 이어집니다.

● PUSH · 자동 브리핑

아침 7시 — 에이전트가 먼저 보고

매일 07:00 (Asia/Seoul) · 무인 자동 실행

정해진 시각에 에이전트가 깨어나, 그날 CEO께 보고될 자료가 쌓인 폴더를 읽고 핵심을 정리해 CEO Teams 1:1 대화방으로 먼저 브리핑을 보냅니다.

  • 지정 폴더의 당일 보고 자료(PDF·PPTX·DOCX)를 읽어 들임
  • 미팅 요약(핵심 5줄) + Talking Point 정리
  • 잘한 점(칭찬) / 짚을 점(지적 사항) 구분 정리
  • BLUF(한 줄 핵심) 우선의 임원 보고체 Teams 메시지
● PULL · 이어지는 대화

그 이후 — 부르면 종합해서 답변

상시 · CEO 질문 시 응답

브리핑을 확인한 CEO가 같은 대화방에서 추가로 궁금한 점을 묻습니다. 에이전트는 당일 보고 자료에 머무르지 않고, 과거 맥락까지 함께 끌어와 답을 이어 갑니다.

  • 당일 보고 자료 + 이전 회의록 + 다른 보고의 자료 종합
  • 여러 출처를 교차 리즈닝해 근거 있는 답변 생성
  • 같은 1:1 채널에서 맥락을 유지한 후속 질의응답
  • 확인이 필요한 항목은 추측 대신 명확히 구분해 보고
HOW IT WORKS

트리거에서 후속 대화까지, 한 흐름

스케줄 트리거가 요약·정리를 거쳐 CEO Teams로 도착하고, 그 자리에서 CEO의 질문이 종합 리즈닝 답변으로 이어집니다.

IMPLEMENTATION OPTIONS

세 가지 구현 방안

같은 시나리오를 세 가지 길로 검토했습니다. 각 방안의 가능 여부, 제약, 그리고 가장 잘 맞는 용도를 함께 정리했습니다.

PLAN A

Copilot Studio (New)

구현 불가

New experience에서 에이전트를 만들고 Workflow로 트리거·발신을 연결하는 정석 경로입니다.

막힌 이유 — 에이전트의 동작 구조가 바뀌면서 Workflow의 커넥터가 정상 동작하지 않습니다. (발신 시 봇 신원이 해소되지 않아 브리핑 발송 배관이 끊김)
PLAN B

Copilot Studio (Classic)

구현 가능 · 권장

Classic 경험으로 빌드하면 Recurrence 트리거 + Publish로 발신 배관이 정상 해소되어 시나리오가 동작합니다. 지금 바로 운영을 시작할 수 있는 권장 방안입니다.

운영 포인트 — Classic UI가 다소 복잡해 CEO가 직접 수정하기보다 스탭 부서가 에이전트를 개선해 나가는 용도에 적합합니다. (예: 스탭의 Talking Point를 추가 반영해 답변 품질을 다듬기)
컨텍스트 유지 — 후속 대화의 맥락을 이어 가려면 질의·답변을 기록한 .md 히스토리 파일을 폴더에 누적해 둡니다. 매 응답 시 이 기록을 함께 참조해 이전 대화의 컨텍스트를 유지합니다.
PLAN C

Microsoft Scout

구현 가능 · 데모 후 검토

Teams와 연결된 Scout Bot을 통해 커뮤니케이션하는 방식. CEO가 직접 동작을 수정하기에도 용이한 근미래형 구성입니다.

검토 포인트 — 에이전트를 상시 구동할 별도 시스템이 필요합니다(VPC 등 사내 인프라 활용 가능). 데모를 먼저 보신 뒤 도입을 검토해 보시면 좋습니다.
비교 항목 Plan A · Studio (New) Plan B · Studio (Classic) Plan C · Scout

권장 — 지금은 Plan B(Classic)로 스탭 부서가 운영하며 에이전트 품질을 키워 나가는 것을 권장합니다. Plan C(Scout)는 데모를 보신 뒤 도입을 검토해 보시면 좋습니다.
